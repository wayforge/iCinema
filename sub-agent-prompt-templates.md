# Sub-Agent 中文 Prompt 模板

这份模板面向 Codex 的日常 `sub-agent` 使用场景。核心原则只有三条：

- 明确拆分维度
- 限定每个 sub-agent 的职责边界
- 要求主线程最后统一汇总

## 通用模板

```text
请把这个任务拆成 {N} 个 sub-agent 并行处理，分别负责：
1. {角色1}
2. {角色2}
3. {角色3}

要求：
- 每个 sub-agent 只关注自己的职责，不要重复别人的工作
- 如果需要读代码，先定位相关文件和调用链
- 如果需要改代码，先给出最小改动方案
- 所有 sub-agent 完成后，你再统一汇总
- 汇总时按“发现 / 证据 / 建议”输出
```

## PR 审查专用

```text
请对当前分支相对 main 的改动启动 4 个 sub-agent 并行审查，分别负责：
1. 正确性和潜在 bug
2. 安全风险
3. 测试缺口和回归风险
4. 可维护性和代码质量

要求：
- 每个 sub-agent 只输出自己负责维度里的问题
- 先找证据，再下结论
- 不要重复同一个问题
- 等全部完成后，再统一汇总成一个审查报告
- 报告按严重级别排序，并附文件路径和原因
```

## Android 开发专用

```text
请启动 3 个 sub-agent 并行分析这个 Android 开发任务：
1. 梳理当前 feature 的 UI、ViewModel、BizPort、Repository 调用链
2. 查项目里已有类似功能是如何实现和组织的
3. 评估最小实现方案、改动范围和验证方式

背景：
{这里写 feature 名、需求说明、相关模块}

要求：
- 先分析现状，不要立刻改代码
- 优先复用现有模式，不重新设计一整套结构
- 每个 sub-agent 输出“关键文件 + 结论 + 建议”
- 最后统一汇总成一个实施方案，并列出推荐的改动顺序
```

## 前端页面实现专用

```text
请启动 3 个 sub-agent 并行处理这个前端任务：
1. 分析页面结构、组件拆分和状态边界
2. 检查现有代码库里可复用的组件和样式模式
3. 评估实现风险、交互细节和适配点

任务目标：
{这里写页面需求 / 设计稿说明}

要求：
- 先分析，不直接写代码
- 每个 sub-agent 输出“建议方案 + 依据文件”

```

## Bug 排查专用

```text
请把这个 bug 排查任务拆成 3 个 sub-agent 并行处理：
1. 复现问题并确认触发条件
2. 定位相关代码路径和根因
3. 评估最小修复方案及可能影响面

背景：
{这里写 bug 描述和复现步骤}

要求：
- 先排查，不要直接大改
- 每个 sub-agent 给出明确结论和证据
- 如果结论不确定，要标出不确定点
- 等都完成后，你再给我一个综合判断和修复建议
```

## 修复加验证专用

```text
请启动 3 个 sub-agent 并行处理：
1. 定位问题根因
2. 准备最小修复方案
3. 设计验证步骤和回归检查项

要求：
- 第 1 个 agent 只做定位
- 第 2 个 agent 基于定位结果提出修复
- 第 3 个 agent 只关注验证和回归风险
- 完成后你统一给出最终实施方案
- 先不要直接提交大范围改动
```

## 批量任务专用

```text
请把下面这些对象分别交给独立 sub-agent 处理，并行执行，最后统一汇总：
- {对象1}
- {对象2}
- {对象3}
- {对象4}

每个 sub-agent 的任务：
- 读取对应对象的相关文件
- 输出主要问题、风险和建议
- 不要处理其他对象

最后汇总时，请做一个对比表。
```

## 最短通用版

```text
请把这个任务拆成多个 sub-agent 并行处理。
一个负责定位代码路径，一个负责找风险，一个负责给最小实现或修复方案。
每个 agent 只做自己的部分，不重复。
全部完成后你再统一汇总，并给出建议的下一步。
```

## 固定增强句式

下面这些句子可以按需拼到任何模板后面：

```text
- 先分析，再实施
- 只关注自己的职责，不要和其他 sub-agent 重复
- 结论必须附证据
- 不确定的地方明确标记
- 全部完成后再统一汇总
- 优先给最小可行方案
- 不要改无关代码
```

## 使用建议

- 小任务不要强行拆 sub-agent，拆了反而更慢
- 适合并行的维度包括：定位、风险、实现方案、验证方案
- 如果你要稳定结果，提示词里一定要写清楚职责边界和汇总格式
- 如果任务要改代码，先让 sub-agent 分析，再决定是否进入实施
